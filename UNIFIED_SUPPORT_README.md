# 🎯 Единый скрипт настройки системы поддержки SubManager

## 📋 Описание

**UNIFIED_SUPPORT_SETUP.sql** — это единый SQL скрипт, который полностью настраивает систему поддержки для приложения SubManager. Запустите его **один раз**, и все будет работать!

## ✨ Что делает этот скрипт?

### 1. **Создание таблицы support_messages**
- ✅ Проверяет, существует ли таблица
- ✅ Создает таблицу с правильной структурой (если её нет)
- ✅ Обновляет существующую таблицу (если её структура устарела)
- ✅ **НЕ УДАЛЯЕТ** существующие данные!

### 2. **Helper функция против рекурсии**
- ✅ Создает функцию `is_user_parent_message()` с `SECURITY DEFINER`
- ✅ Предотвращает бесконечную рекурсию в RLS политиках
- ✅ Оптимизирует проверку прав доступа к сообщениям

### 3. **Row Level Security (RLS) политики**

#### SELECT политика
- 👤 Пользователи видят свои сообщения
- 💬 Пользователи видят ответы админов на их сообщения
- 👑 Владельцы видят ВСЕ сообщения

#### INSERT политика
- 👤 Пользователи могут создавать свои сообщения
- 👑 Владельцы могут создавать любые сообщения (включая ответы)

#### UPDATE политика
- 👤 Пользователи могут редактировать свои сообщения в течение 5 минут
- 📖 Пользователи могут помечать ответы админов как прочитанные (`is_read`)
- 👑 Владельцы могут обновлять любые сообщения

#### DELETE политика
- 👤 Пользователи могут удалять свои сообщения в течение 5 минут
- 👑 Владельцы могут удалять любые сообщения

### 4. **Поле is_read**
- ✅ Отслеживание статуса прочтения сообщений
- ✅ Пользователи могут помечать админ-ответы как прочитанные
- ✅ Владельцы могут помечать сообщения пользователей как прочитанные

### 5. **Оптимизация производительности**
Создает 6 индексов:
- `support_messages_user_id_idx` — быстрый поиск по пользователю
- `support_messages_created_at_idx` — сортировка по времени
- `support_messages_is_admin_reply_idx` — фильтрация админских ответов
- `support_messages_parent_id_idx` — поиск связанных сообщений
- `support_messages_is_read_idx` — фильтрация непрочитанных
- `support_messages_user_is_read_idx` — составной индекс для сложных запросов

## 🔧 Решение проблем

### Проблема: "Infinite recursion detected"

**Решение:** Этот скрипт уже содержит исправление! Helper функция `is_user_parent_message()` использует `SECURITY DEFINER`, который минует RLS и предотвращает рекурсию.

### Проблема: "Permission denied for table support_messages"

**Решение:** Убедитесь, что:
1. RLS включен (скрипт делает это автоматически)
2. Политики созданы (проверьте вывод скрипта)
3. Вы залогинены как аутентифицированный пользователь

### Проблема: "Column is_read does not exist"

**Решение:** Запустите скрипт — он автоматически добавит недостающее поле.

## 🎓 Для разработчиков

### Как работает защита от рекурсии?

Старая проблемная политика:
```sql
-- ❌ ПЛОХО: Вызывает рекурсию!
is_admin_reply = true 
AND parent_message_id IN (
    SELECT id FROM support_messages WHERE user_id = auth.uid()
    -- Здесь снова применяется RLS → рекурсия!
)
```

Новая исправленная политика:
```sql
-- ✅ ХОРОШО: Использует helper функцию
is_admin_reply = true 
AND parent_message_id IS NOT NULL
AND public.is_user_parent_message(parent_message_id, auth.uid())
-- Helper функция с SECURITY DEFINER минует RLS!
```

### Почему SECURITY DEFINER безопасен?

- Функция выполняется с правами создателя (owner базы данных)
- Минует RLS только внутри функции
- Возвращает только boolean (безопасно)
- Не раскрывает данные других пользователей
- Предотвращает бесконечную рекурсию
