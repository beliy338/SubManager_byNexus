## 📊 Сравнение подходов

### ❌ Старый подход (с рекурсией)

```sql
CREATE POLICY "Users can view messages"
    ON support_messages
    FOR SELECT
    USING (
        user_id = auth.uid()
        OR parent_message_id IN (
            -- ⚠️ Подзапрос к той же таблице!
            SELECT id FROM support_messages 
            WHERE user_id = auth.uid()
        )
    );
```

**Проблема:** PostgreSQL пытается применить ту же политику к подзапросу → рекурсия!

---

### ✅ Новый подход (без рекурсии)

**Шаг 1: Создать функцию**
```sql
CREATE FUNCTION is_user_parent_message(parent_id UUID, user_uuid UUID)
RETURNS BOOLEAN
LANGUAGE plpgsql
SECURITY DEFINER  -- ← Обходит RLS!
AS $$
BEGIN
    RETURN EXISTS (
        SELECT 1 FROM support_messages 
        WHERE id = parent_id AND user_id = user_uuid
    );
END;
$$;
```

**Шаг 2: Использовать в политике**
```sql
CREATE POLICY "Users can view messages"
    ON support_messages
    FOR SELECT
    USING (
        user_id = auth.uid()
        OR (
            is_admin_reply = true
            AND is_user_parent_message(parent_message_id, auth.uid())
            -- ✅ Функция вместо подзапроса!
        )
    );
```

**Решение:** Функция работает с правами владельца → нет рекурсии!
