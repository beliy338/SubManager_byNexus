# Исправление чата поддержки

## Проблемы
1. ❌ **Ошибка "permission denied for table users"** - таблица создана с неправильным внешним ключом
2. ❌ Простые пользователи не могут отправлять сообщения
3. ❌ При открытии чата выдается ошибка загрузки сообщений

## ⚠️ ВАЖНО: Причина ошибки

Ошибка `permission denied for table users` возникает из-за внешнего ключа:
```sql
user_id UUID NOT NULL REFERENCES auth.users(id)
```

Postgres пытается проверить существование пользователя в `auth.users`, но к этой системной таблице нет доступа через RLS. 

**Решение:** Убрать внешний ключ и хранить `user_id` как обычное UUID поле.

1. Откройте **Supabase Dashboard** → SQL Editor
2. Выполните **`recreate-support-messages-table.sql`**

Этот скрипт:

## Что было исправлено в коде

### 1. Упрощена политика INSERT
**Было:**
```sql
WITH CHECK (user_id = auth.uid() OR ...)
```

**Стало:**
```sql
WITH CHECK (true)
```

Теперь все аутентифицированные пользователи могут отправлять сообщения.

### 2. Улучшена обработка ошибок в SupportModal
- Добавлено подробное логирование ошибок
- Специальные сообщения для разных типов ошибок
- Проверка авторизации перед отправкой
- Graceful fallback если таблица не создана

### 3. Исправлен текст приветствия
**Было:** "Опишите ваш вопрос/проблему"
**Стало:** "Опишите Ваш вопрос/проблему"

## Сохранение диалогов

Диалоги **автоматически сохраняются** в базе данных Supabase:
- При отправке сообщения оно сохраняется в `support_messages`
- При открытии чата загружаются все сохраненные сообщения
- При перезапуске приложения все сообщения остаются

Есл�� сообщения не сохраняются, проверьте:
1. Таблица создана в Supabase
2. RLS политики настроены правильно
3. Пользователь авторизован

## Диагностика проблем

Откройте консоль браузера (F12) и попробуйте отправить сообщение.

### Если таблица не создана:
```
Error code: 42P01
Message: relation "public.support_messages" does not exist
```
**Решение:** Выполните `supabase-support-messages-table.sql`

### Если нет прав доступа:
```
Error code: 42501
Message: new row violates row-level security policy
```
**Решение:** Выполните `fix-support-messages-policies.sql`

### Если пользователь не авторизован:
```
Error: Вы не авторизованы
```
**Решение:** Выйдите и войдите заново

## Структура базы данных

### Таблица support_messages:
```
id                  UUID (PRIMARY KEY)
user_id             UUID (БЕЗ внешнего ключа - просто UUID)
user_email          TEXT
user_name           TEXT
message             TEXT
is_admin_reply      BOOLEAN (false для обычных, true для владельцев)
parent_message_id   UUID (для ответов на сообщения)
created_at          TIMESTAMP
```

### RLS Политики:

1. **SELECT:** Пользователи видят свои сообщения, владельцы видят все
2. **INSERT:** Все аутентифицированные пользователи могут отправлять
3. **UPDATE:** Свои сообщения в течение 5 минут, владельцы - любые
4. **DELETE:** Свои сообщения в течение 5 минут, владельцы - любые
