# Инструкция по активации системы владельцев

## Шаг 1: Создание таблицы services в Supabase

1. Откройте ваш проект в [Supabase Dashboard](https://supabase.com/dashboard)
2. Перейдите в раздел **SQL Editor** (левое меню)
3. Нажмите **"New Query"**
4. Скопируйте содержимое файла `supabase-services-table.sql` и вставьте в редактор
5. Нажмите **"Run"** или нажмите `Ctrl+Enter`
6. Дождитесь сообщения об успешном выполнении

## Шаг 2: Проверка создания таблицы

1. В Supabase Dashboard перейдите в **Table Editor**
2. Найдите таблицу **services** в списке таблиц
3. Проверьте структуру таблицы:
   - id (uuid, primary key)
   - user_id (uuid, foreign key)
   - name (text)
   - category (text)
   - description (text, nullable)
   - icon (text, nullable)
   - is_popular (boolean)
   - pricing_plans (jsonb)
   - created_at (timestamp)
   - updated_at (timestamp)

## Шаг 3: Проверка Row Level Security (RLS)

1. В Table Editor выберите таблицу **services**
2. Нажмите на иконку **замка** или перейдите в **"RLS Policies"**
3. Убедитесь, что созданы следующие политики:
   - ✅ "Anyone can view services"
   - ✅ "Only owners can insert services"
   - ✅ "Only owners can update services"
   - ✅ "Only owners can delete services"

## Шаг 4: Проверка триггеров

1. Перейдите в **Database** → **Triggers**
2. Убедитесь, что созданы триггеры:
   - ✅ `update_services_updated_at_trigger` - автоматически обновляет updated_at
   - ✅ `cascade_delete_service_subscriptions` - каскадно удаляет подписки пользователей

## Возможные проблемы и решения

### Ошибка: "permission denied for table services"

**Причина**: RLS политики не применены или неправильно настроены

**Решение**:
1. Перейдите в SQL Editor
2. Выполните команду:
```sql
ALTER TABLE public.services ENABLE ROW LEVEL SECURITY;
```
3. Повторно выполните все политики из файла `supabase-services-table.sql`

### Ошибка: "relation services does not exist"

**Причина**: Таблица не создана

**Решение**:
1. Повторно выполните весь скрипт из `supabase-services-table.sql`

### Владелец не может создавать сервисы

**Причина**: Email в системе не совпадает с указанным в политиках

**Решение**:
1. Проверьте email, с которым вы вошли в систему
2. Убедитесь, что это `max.sokolvp@gmail.com` или `belovodvadim@gmail.com`
3. При необходимости обновите политики в Supabase

### Подписки пользователей не удаляются при удалении сервиса

**Причина**: Триггер не создан или неактивен

**Решение**:
1. В SQL Editor выполните:
```sql
-- Проверка существования триггера
SELECT * FROM pg_trigger WHERE tgname = 'cascade_delete_service_subscriptions';
```
2. Если триггер не найден, повторно выполните часть скрипта с созданием триггера
